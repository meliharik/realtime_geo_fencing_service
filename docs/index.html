üì° API Endpoints

POST /api/geofencing/check

Check GPS location for violations. Returns violation details if point is inside any no-parking zone.

GET /api/geofencing/check-quick

Quick check with query parameters: ?scooterId=SC-001&lat=37.78&lon=-122.41

GET /api/geofencing/zones

Get all active no-parking zones from database.

GET /api/geofencing/cache/stats

Get Redis cache statistics and health metrics.

POST /api/geofencing/cache/refresh

Manually refresh Redis cache from database.

GET /api/geofencing/violations/{scooterId}

Get violation history for a specific scooter.

üîå WebSocket API

Connect to WebSocket

const socket = new SockJS('http://localhost:8080/ws/gps-stream'); const stompClient = new StompJs.Client({ webSocketFactory: () => socket });

Subscribe to Topics

// Subscribe to violation alerts stompClient.subscribe('/topic/alerts', (message) => { const alert = JSON.parse(message.body); console.log('Violation detected:', alert); }); // Subscribe to system status stompClient.subscribe('/topic/status', (message) => { console.log('System status:', message.body); });

Send GPS Events

stompClient.publish({ destination: '/app/gps', body: JSON.stringify({ scooterId: 'SC-001', latitude: 37.7800, longitude: -122.4150, timestamp: new Date().toISOString() }) });

üèóÔ∏è Architecture

The system uses a cache-aside pattern with automatic fallback:

  1. GPS Event Received - Via REST API or WebSocket
  2. Cache Check - Query Redis for cached zones (~0.1-0.5ms)
  3. In-Memory Spatial Query - JTS point-in-polygon check
  4. Database Fallback - PostGIS spatial query if cache miss (~5-10ms)
  5. Rate Limiting - Check for duplicate violations
  6. Persist & Alert - Save to database and broadcast via WebSocket
